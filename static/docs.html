flakybin · docs

Deterministic HTTP chaos for testing uptime monitors and retry logic.

flakybin is a fault-injection server that fails on a deterministic, seed-reproducible schedule: it returns errors, hangs, and drops connections during scheduled outage windows, emulating a service that goes down on a repeating cycle. Outage placement is computed purely from the request URL, so the same parameters always yield the same outage windows on the wall clock, across restarts and across machines. No state, no database. Point an uptime monitor at it and get a known-good failure pattern to test against.

Try the interactive schedule explorer to visualize and tune a schedule, then copy a failure URL.

Common parameters

Every failure endpoint accepts these query parameters:

parammeaning
periodlength of one repeating cycle, e.g. 24h, 1h, 10m (default 24h)
durationlength of a single outage, e.g. 5m (default 15m)
countnumber of outages per period — supply this or uptime (not both)
uptimetarget availability as a percent, e.g. 99.9; converted to an outage count (at least one, at most one short of full). Defaults to 90 when neither count nor uptime is given. 100 is rejected with 400
seedRNG seed (default 0); change it to reshuffle the outage placement

Modes

The first path segment selects how outages are placed within each period:

modebehavior
even (default)count outages of length duration spaced uniformly across the period (one per period/count interval); the seed shifts the whole pattern's phase. No divisibility constraint; regular and predictable.
jittercount outages of length duration are dropped at seed-hashed offsets anywhere within the period. No divisibility constraint; looks like organic flapping.
noiseThe period is divided into N = period/duration fixed slots; count of them are outages. period must divide evenly by duration (otherwise 400). Grid-aligned and easy to reason about.

Endpoints

routeduring an outage
GET /{mode}/status/{code}responds with HTTP {code} (e.g. 503, 429, 500); 200 when up. Optional retry-after=auto or retry-after=<seconds> sets the Retry-After header.
GET /{mode}/hangwithholds the response until the outage ends, or for for=<duration> if given. Emulates a timeout.
GET /{mode}/dropwrites after=<bytes> of body (default 0) then resets the TCP connection. Emulates a mid-flight drop.
GET /{mode}/inspectshows the schedule and its outage windows. A browser (Accept: text/html) gets a graphical timeline; scripts get JSON. Force either with ?format=html / ?format=json. The JSON carries a failure_links block with ready-to-call URLs for every status code plus hang and drop. Triggers no failure.
GET /diceroll 🎲redirects (302) to inspect for a random but valid schedule — random mode, period, duration, uptime, and seed.

Examples

# Even (default): a 24h period with outages totalling ≈ 99% availability,
# evenly spaced, returning 503 with an automatic Retry-After during each outage.
/even/status/503?period=24h&duration=5m&uptime=99&retry-after=auto

# Ten 1-minute blips scattered through each hour, seed 42 (jitter).
/jitter/hang?period=1h&duration=1m&count=10&seed=42

# Connection drops after 128 bytes, once per 30-minute period.
/jitter/drop?period=30m&duration=2m&count=1&after=128

# Noise: a 24h period cut into 5-minute slots, 12 of them down (≈ 99.58% uptime).
/noise/status/503?period=24h&duration=5m&count=12&retry-after=auto

# Noise, first-N-minutes-of-the-hour style: 1h period, 10m outage, 1 slot.
/noise/status/500?period=1h&duration=10m&count=1

# Inspect the schedule without triggering it.
/even/inspect?period=24h&duration=5m&uptime=99

Response headers

Every response advertises the resolved schedule and current state via X-Flaky-* headers: Mode, Period, Duration, Count, Seed, Uptime, In-Outage, and either Outage-Ends or Next-Outage (RFC 3339).
